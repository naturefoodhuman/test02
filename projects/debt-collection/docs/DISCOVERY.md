# DISCOVERY —— 个人合法讨债助手（试点项目 / 工厂陪练）

> ⚠️ 元说明：本项目是 FORGE Factory 的**陪练沙包**，核心目的是压测工厂、摸清能力边界（见 HANDOFF 0.0）。
> 产物质量按真项目标准做，但成败标准看"工厂这一阶段表现如何"。

## 0. 一句话定位
为个人债权人（老板，涉 7~8 笔混合性质债务）提供一个**本地运行、合法合规**的工具：
管理多笔债务台账，并为每笔债务生成"**最佳合法讨债策略报告**"。
**红线（D-007/法律）**：系统只走合法路径，绝不自行非法查询他人财产/隐私。

## 1. 核心用户场景（≥3）
- **场景 A · 录入与台账**：老板把一笔债务的关键信息（债务人、金额、借据/合同、时间线、已还情况、债务性质）录入，系统结构化保存为台账，一眼看清 7~8 笔的全局（总额、诉讼时效紧迫度、阶段）。
- **场景 B · 策略报告**：针对某一笔，系统产出"最佳讨债策略报告"——含：①证据链体检（够不够立案/胜诉）②诉讼时效预警 ③讨债阶梯（协商→律师函→支付令→起诉→执行）④各阶段话术/文书要点 ⑤**合法**财产线索查询清单（失信网/裁判文书网/工商公示/不动产等公开渠道 + 何时该委托律师申请调查令）。
- **场景 C · 时效与节点提醒**：系统根据每笔债务的关键日期（最后还款/最后催收/诉讼时效到期）提示"该走下一步了"，避免因拖延丧失胜诉权。
- **场景 D · 分性质处理**：民间借贷 vs 买卖合同欠款，举证要点和法律路径不同，系统按债务性质给差异化建议。

## 2. 同类产品 / 开源项目清单（≥3）
- **商业法律 AI**：通义法睿、各类"法律大模型"问答（通用，不做个人多笔债务台账+策略闭环）。
- **催收科技（To B）**：得助智能等 AI 质检/智能外呼系统——面向催收机构合规，**不是给个人讨债用**。
- **法律服务平台**：华律网/66law 等在线咨询（人工问答，非自动化策略报告）。
- **执行信息公开渠道**：中国执行信息公开网、裁判文书网、国家企业信用信息公示系统（这些是数据源，不是产品）。
- **结论**：个人向、本地化、多笔债务"台账+合法策略闭环"的工具是空白点，有真实价值。

## 3. 已知失败案例（≥1，避坑）
- **暴力/软暴力催收入刑**：2026 新规，电话轰炸、爆通讯录、上门蹲守等 8 类行为直接判刑；北京朝阳法院判 31 名催收员寻衅滋事罪。→ 系统**绝不能**生成任何施压/骚扰债务人或第三方的功能。
- **非法查询个人信息入刑**：个人擅自通过"调查机构/网络工具"查他人银行流水、隐藏财产、身份证号 → 可能构成侵犯公民个人信息罪（最高 7 年）。→ 系统**绝不能**自行扒数据，只能指引合法渠道。
- **拖延丧失时效**：现实中大量债权人因超过诉讼时效（一般 3 年）败诉。→ 系统必须做时效预警。

## 4. 非功能要求
- **合规（最高优先级）**：所有功能在中国大陆现行法律框架内；涉及查询一律走公开渠道或律师调查令路径。
- **隐私/安全**：债务数据是高度敏感个人信息，**纯本地存储**（SQLite/本地文件），不上传云端；可考虑加密。
- **成本**：策略生成优先本地模型（qwen3.6-35b / local/coder），复杂法律推理可选 GLM；零强制云端成本。
- **可用性**：面向非技术用户（老板本人），输出强制中文、人话、可直接照做。

## 5. 验收标准草案（描述性，SPEC 阶段转测试）
- AC-1：能录入并持久化 ≥8 笔债务，每笔含必要字段（债务人、金额、性质、关键日期、证据清单、当前阶段）。
- AC-2：对任一笔能生成结构化"策略报告"，包含第 1 节场景 B 的 5 个要素。
- AC-3：能对每笔计算并提示诉讼时效剩余时间 / 是否临近/超期。
- AC-4：报告中所有"查财产/查信息"建议均为合法渠道，无任何非法或施压性内容（合规自检）。
- AC-5：纯本地运行，无敏感数据外传。

## 6. 主动识别的空白 / 矛盾（DISCOVERY 阶段核心价值）
- **矛盾1（已解决）**：老板要"查到欠款人名下财产" vs 个人无权查 → 重定义为"指引合法查询路径"。这是工厂在本阶段成功拦截的违法红线。
- **空白1**：策略质量依赖法律知识准确性。LLM 可能产生"法律幻觉"（编造法条/程序）→ SPEC 阶段需设计"法律知识来源/免责声明/建议人工复核"机制。
- **空白2**：债务的关键日期、证据细节需老板提供，系统无法替他取证 → 录入引导要清晰。
- **待定**：是否需要生成可直接用的文书模板（借条补强、律师函、起诉状要点）？倾向 V1 给"要点+模板片段"，不做正式法律文书生成（避免误用风险）。

## 7. 【工厂压测观察】本阶段对工厂能力的检验记录
> 这部分是陪练的真正产出——记录工厂在 DISCOVERY 阶段的表现。
- ✅ **违法红线识别能力**：工厂（经 Agent）主动调研并拦截了"个人非法查财产"这一刑事红线，重定义为合法方向。这是 DISCOVERY 阶段最该有的能力，验证通过。
- ✅ **结构化提问能力**：通过 ask_user 用 3 组选项把模糊需求（"高效要回债"）收敛为可执行边界。
- ✅ **调研能力（FR-001-3）**：完成同类产品 + 已知失败案例调研。
- 🟡 **待观察**：进入 SPEC 后，本地模型能否产出靠谱的架构/法律逻辑设计（将对比 local/primary vs GLM）。
- 📌 **改进点候选**：DISCOVERY 全靠 Agent 手动驱动，工厂暂无"自动检查 DISCOVERY.md 字段完整性"的脚本（架构书 Exit Gate 要求自动校验）。→ 记入工厂改进 backlog。

## 8. 【第10轮】老板补充意见（6 点）——已逐条采纳

> 老板已同意前述基本要点，以下为补充的不同意见与想法。其中 ⭐ 标记的是"应升级为工厂通用能力"的项。

### 要点1 · 基本要点全部同意 ✅
- 前 7 节内容（含违法红线重定义、V1 范围）老板确认同意。

### 要点2 ⭐ · 数据能力是核心，升级为工厂通用能力
- 数据的**收集、整理、提炼总结、经验复用**能力十分关键 → 升级为**各项目通用能力**（不只讨债）。
- 需包含：**多数据源权重分配** + **数据质量把控**（判别渠道内容是"干货"还是"夹带私货/软文/营销"）。
- **【第11轮调研结论】**（详见 `docs/research/data-acquisition-feasibility.md` 第二部分）：
  业界公认做法 = 来源分级(source tiering) + 多源交叉验证 + Faithfulness 忠实度校验(防幻觉) + Reranker 重排 + 去软文 + 黑白名单。
- **解决老板"能定性打标但无法量化权重"的痛点**：老板只给**定性等级 + 黑名单**（律所实务>自媒体、某博主不可信），
  系统把定性映射为量化权重（如 官方1.0 / 律所0.8 / 媒体0.5 / 自媒体0.3 / 黑名单0.0），**量化的活交给系统**。
- 落地：`_factory/skills/data-quality.skill.md` + 每项目一个可编辑的 `sources.yaml`（来源等级/黑名单）。
- 记入工厂改进 backlog（FB-5）。

### 要点3 ⭐ · 本地模型的"联网取数"能力必须明确供给方式
- 本地 Ollama 模型**本身不具备联网检索能力**，必须由工厂在外层提供"取数能力"。
- **【第11轮调研结论】**（详见 `docs/research/data-acquisition-feasibility.md`）：
  - browser-use 等浏览器 Agent 通用任务成功率高（WebVoyager 89.1%），可接本地/GLM 模型；
    但对**强风控平台（企查查/天眼查/微信/小红书）基本爬不动**，且账号自动抓**封号风险真实存在 + ToS 合规风险**。
  - **好消息**：讨债最关键的数据（失信被执行人、执行公开、裁判文书、工商 gsxt）**恰好都在低风险的官方公开渠道**。
  - **采纳的取数三层方案**（默认走低风险，高风险需老板逐次授权）：
    - L1 公开官方渠道直取（默认，首选）；
    - L2 半自动"人在环"（browser-use 接 GLM，遇验证码/登录就停下等老板手动过）；
    - L3 账号自动抓（**默认关闭**，仅 L1/L2 不足时、需老板逐次显式授权；账号安全优先于数据，不赌账号）。
  - **明确建议**：企查查/天眼查若必须用，走**官方付费 API**而非账号爬虫。
- 老板授权（带硬约束，已记录）：可提供个人账号，但**账号安全 > 数据**；系统须限频/拟人化/失败即停，且复核 ToS 合规。
- 记入工厂改进 backlog（FB-6：取数抽象层，分 L1/L2/L3）。

### 要点4 · 讨债项目的核心成功标准 = 实际落地讨回率（不是方案花哨）
- **唯一硬指标 = 最终能否把钱要回来**，不是方案看起来多专业。
- **强约束**：不能动辄"写催收函/写诉状"这类**脱离国内实际执行环境**的纸面方案。
- 必须正面应对**执行难**：避免"债务人明明有资产、判了也多年执行不到（老赖）"的死局。
- → V1 策略报告必须包含"**执行可行性预判**"：先评估"赢了官司能不能真执行到财产"，
  把"诉前财产保全""有无可执行财产线索""对方失信/规避执行历史"前置到策略里，而非走完诉讼才发现执行不了。
- → 验收标准升级：新增 **AC-6：每笔策略报告必须含"执行可行性评估 + 回款概率判断"，并优先非诉/保全等高回款率手段。**

### 要点5 · 需求讨论阶段要更充分（不急于推进）
- 老板要求：**提升需求讨论的丰富度和深度，不担忧耗时，充分讨论、多参考多研究后再启动后续**。
- → 工厂流程调整：**本项目 DISCOVERY 阶段延长**，不急着过 Gate-1 进 SPEC；继续深挖。
- → 工厂改进（FB-7）：DISCOVERY 阶段增加"深度自检清单"（是否调研充分、是否多源对比、是否穷尽边界），
  防止 Agent 急着推进。这也修正了之前"急于产出 DISCOVERY.md 就想进 SPEC"的倾向。

### 要点6 ⭐ · 工厂需具备通用的本地资料整理能力
- 本地资料含**图片 / PDF / 通话录音**等，目标=统一转为**结构化格式**。
- 已完成横向对比（见 `docs/research/ingestion-tools-comparison.md`）：
  - **文档/PDF/图片**：主力 MinerU（中文强）+ 轻量入口 MarkItDown（MIT/多格式）；备选 Marker/Docling。
  - **通话录音**：FunASR（中文最强、本地、内置说话人分离）。
- → 工厂改进（FB-8）：新增通用 **Ingestion 能力层**（skill + 可运行 Pattern），输出结构化 Markdown/JSON，
  对接已配的 `local/embedding`(bge-m3) 做向量化。讨债项目的借据照片、合同 PDF、催收通话录音正好是首个用例。
