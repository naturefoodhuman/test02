# 工厂核心升级：决策大脑 / 专家系统（Expert System）设计

> 老板第23轮提出：工厂的"专家咨询能力"要独立出来，成为**所有项目可复用的领域专家能力**。
> 例：购物助手=育儿专家+婴儿用品专家+购物能手；医疗助手=顶级医生团队+医生猎头。
> 核心理念：**专家能力 ≠ 仅靠 LLM**，而是 LLM + 沉淀到本地的领域知识（书籍/指南/专家共识/行业规范/顶级论文/论坛报告/优质博客）+ 持续更新。
> ⚠️ R6 红线：关键知识获取不到，必须征询老板代为下载，绝不假装。

## 一、为什么需要独立的专家系统（而非每个项目各写一套）
- 讨债项目已暴露：它本质需要一个"债务律师专家"（法条+执行实务+本地口径）。
- 未来项目（购物/医疗/...）都需要各自领域专家。**重复造=浪费 + 质量不一**。
- → 把"专家"抽象为工厂级可复用资产：**一个专家 = 角色设定 + 领域知识库(RAG) + 模型路由 + 更新机制**。

## 二、一个"专家"的构成（四件套）
```
专家(Expert) = ①角色画像 + ②领域知识库 + ③模型路由 + ④更新机制
┌────────────────────────────────────────────────────────────┐
│ ① 角色画像 expert.yaml：身份/专长/思维方式/禁忌/输出规范      │
│ ② 领域知识库 knowledge/：                                    │
│    - 书籍/指南/专家共识/行业规范/顶级论文/论坛报告/优质博客    │
│    - 经 ingestion(FB-8) 结构化 → embedding 向量化 → RAG 检索 │
│    - 每条带 sources.yaml 可信度分级(FB-5) + 更新日期         │
│ ③ 模型路由：该专家用哪个模型最合适(可不同于工厂默认)         │
│    - 如医疗用最强推理模型；中文实务用 Qwen3.5/GLM           │
│ ④ 更新机制：知识定期刷新 + 缺口清单(R6 缺啥求助老板)         │
└────────────────────────────────────────────────────────────┘
```

## 三、目录结构（工厂级）
```
_factory/experts/
├── _TEMPLATE.expert/                 # 专家模板(复制创建新专家)
│   ├── expert.yaml                   # 角色画像 + 模型路由 + 知识来源清单
│   ├── knowledge/                    # 沉淀的领域知识(结构化后)
│   │   ├── _sources.yaml             # 本专家的知识来源可信度分级(复用FB-5)
│   │   ├── _gaps.md                  # ⭐知识缺口清单(R6：缺什么/为何重要/是否需老板下载)
│   │   └── *.md / *.json             # 结构化知识(经 ingestion)
│   └── README.md
├── debt-lawyer.expert/               # 首个专家：债务律师(从 debt-collection 抽象)
└── （未来）child-care.expert / shopping.expert / medical.expert ...
```

## 四、专家系统与已有能力的复用关系（不重造）
- **知识入库** → 复用 `ingestion-pipeline`(FB-8) 把书籍/PDF/论文转结构化。
- **来源可信度** → 复用 `data-quality` + `sources.yaml`(FB-5)：书籍/指南=高，论坛/博客=低需交叉验证。
- **向量检索** → 复用 `local/embedding`(bge-m3) 做 RAG。
- **取数更新** → 复用 `data-acquisition`(FB-6) 抓取行业报告/论坛(合规)。
- **模型路由** → 复用 LiteLLM 网关；专家可指定更强模型(见下"模型选型")。
- **遥测** → 复用 `llm-telemetry`(R4) 记录每次专家咨询。
- → **专家系统 = 把已有工厂能力"编排"成领域大脑**，是工厂能力的集大成。

## 五、多专家协作（老板的"专家团队"场景）
购物助手例：`shopping.expert` 编排 `child-care.expert` + `baby-product.expert` + `deal-hunter.expert`。
- 实现为"专家会诊"：主专家拆解问题 → 各子专家用各自知识库回答 → 主专家综合。
- 类似讨债的"动态博弈"，是一种**编排模式(Pattern)**，可复用。

## 六、本机可用的更强开源模型（R5：这次调研穷尽，补此前不足）
> M1 Max 64GB 可跑的、可能优于当前 qwen3.6-35b 的候选（按需 ollama 拉取）：

| 模型 | 规模(本机可行性) | 强项 | 适合做哪类专家脑 |
|---|---|---|---|
| **Qwen3.5-35B-A3B** | MoE,激活3B,Q4 可跑 | 中文第一/全能,超越旧235B | 通用/中文实务专家(债务律师) |
| **Qwen3.5-27B(密集)** | 16-24GB,M系列可跑 | SWE 72.4%,推理强 | 编程/逻辑专家 |
| **DeepSeek-V3.2 蒸馏/R1** | 蒸馏版本机可跑 | 数学/推理天花板 | 医疗诊断/复杂推理专家 |
| **GLM-Z1-9B / GLM 系列** | 9B 轻松跑 | 中文/Agent | 政务/中文专家 |
| **Devstral Small 2 (24B)** | 32GB Mac 可跑,Apache2.0 | 编程专精 | 代码专家 |
| **GPT-OSS-20B** | ~15GB,Apache2.0 | 科学推理 | 通用推理 |
| **bge-m3 / qwen3-embedding**(已有) | 轻量 | 向量检索 | 所有专家的 RAG 底座 |

> 结论：当前 qwen3.6-35b 够用但非唯一；**不同专家可路由到不同最优模型**。云端兜底仍用 GLM。
> 顶级闭源(Claude Opus/GPT-5/Gemini)只能 Manual Gate 人工介入(C1)，用于最关键决策。
