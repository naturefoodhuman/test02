# 工厂通用能力调研：取数能力（FB-6）+ 数据质量（FB-5）可行性评估

> 来源：老板要点3（取数路线）+ 要点2（数据质量）。先调研同类方案的真实效果与风险，再评估收益。
> 这是**工厂级通用能力**决策依据，不只服务讨债项目。

---

# 第一部分：取数能力（FB-6）—— browser-use 等浏览器 Agent 评估

## 1. browser-use 现状（事实）
- 开源、MIT、90k+ stars、YC W25 出身；0.12 起弃用 Playwright 改用 CDP，token 用量减半。
- WebVoyager 基准成功率 **89.1%**（开源浏览器 Agent 最高），但这是"通用网页任务"，不等于"对抗强风控平台"。
- **接本地/开源模型**：可接任意 OpenAI 兼容模型（实测有人用 Qwen2.5-72B、DeepSeek 跑通）。
  → 理论上可接我们的 `cloud/glm-primary` 或 `local/primary`。⚠️ 但本地 35b 做浏览器 Agent 的可靠性显著低于 GLM/Claude，多步骤任务易失败。

## 2. 真实局限与封号风险（事实，关键）
- **强风控平台基本爬不动**：社区实测——微信/小红书/飞书/豆包等"屏蔽严"，浏览器 Agent 也拿不到；
  高质量内容往往就在这些平台，且需手机验证码登录。
- **企查查/天眼查**：有**字体反爬 + 滑块验证 + 动态IP检测 + 登录态校验**。爬取需要：代理IP池 + 拟人化轨迹 + 打码/OCR + Cookie 维护。
- **封号触发点**（来自反爬经验）：
  - 同账号高频请求 / 异常时段
  - IP 地域频繁切换（"1小时切3省"被标高风险）
  - 机械化滑块轨迹（速度均匀=机器）
  - 缺正常 Cookie/Referer/UA、并发过高
- **结论**：用**老板个人账号 + 自动化登录抓取强风控平台**，**封号风险真实且不低**，且很多平台 ToS 明确禁止爬虫，存在合规风险。

## 3. 风险分级（按数据源）——给老板的收益/风险表

| 数据源 | 取数难度 | 封号/合规风险 | 建议方式 |
|---|---|---|---|
| **失信被执行人网 / 执行信息公开网** | 低（公开查询）| 低 | ✅ 直接抓/官方查询，**首选** |
| **裁判文书网** | 中（有验证）| 低-中 | ✅ 半自动：Agent 辅助 + 人工过验证码 |
| **国家企业信用信息公示系统(gsxt)** | 中 | 低（官方）| ✅ 优先用它，权威 |
| **企查查/天眼查** | 高（强反爬）| **中-高（ToS禁爬+可能封号）** | ⚠️ 优先用其**官方API**；不建议账号自动抓 |
| 微信/小红书等 | 极高 | 高 | ❌ 不做自动抓，必要时人工 |

## 4. 推荐方案（FB-6 工厂取数抽象层）
分三层，**默认走低风险，高风险需老板逐次授权**：
1. **L1 公开渠道直取（默认）**：失信网/执行公开网/裁判文书网/gsxt 等官方公开数据 → 风险低，首选。
2. **L2 半自动（人在环）**：browser-use（接 GLM）驱动，**遇验证码/登录就停下来等老板手动过**，不强行绕过 → 大幅降封号风险。
3. **L3 账号自动抓（默认关闭，需老板逐次显式授权）**：仅在 L1/L2 不够时；
   必做：限频(随机延时1-3s)、拟人化、单账号低频、失败立即停、专用浏览器 profile、**账号安全优先于数据**。
   ⚠️ 仍无法 100% 保证不封号；建议**宁可慢、宁可人工，不赌账号**。
- 工厂落地：抽象成 `_factory/skills/data-acquisition.skill.md` + 统一"取数接口"（项目只声明"要什么数据"，由取数层决定走哪层）。

## 5. 给老板的明确建议
- **不建议**让系统自动登录你的账号去抓企查查/天眼查这类强风控平台（封号风险 + ToS 风险）。
- **建议**：L1 公开官方渠道为主（讨债最关键的"失信/被执行/裁判文书/工商"恰好都在公开渠道）+ L2 人在环半自动。
- 企查查/天眼查若必须用 → 走**官方付费 API**（合规、不封号），而非账号爬虫。
- 这条路虽"慢一点"，但符合要点4（落地成功率优先）——账号被封/数据违法反而拖垮项目。

# 第二部分：数据质量（FB-5）—— 业界公认做法

## 1. 业界怎么做"辨别干货 vs 夹带私货 / 来源可信度"
综合 RAG 评估与事实核查领域（RAGAS / RGB / FEVER / reranker 实践）的公认做法：

| 手段 | 说明 | 能否用上老板的辅助 |
|---|---|---|
| **来源分级（source tiering）** | 给数据源打可信度等级：官方法规/法院 > 律所实务 > 媒体 > 自媒体 | ✅ 老板可定性打标（律所实务>自媒体）+ 提供**负面标签**（某博主不可信）|
| **多源交叉验证** | 同一结论需 ≥2 个独立可信源印证，单源孤证降权 | 自动 |
| **Faithfulness 校验** | 生成内容必须能从来源推导，防幻觉（RAGAS 忠实度，与人判一致性达 0.95）| 自动（可用 GLM 做裁判）|
| **Reranker 重排** | 检索后用重排序模型按相关性+质量排序，过滤噪声 | 自动（可配重排模型）|
| **去营销/去软文** | 规则+模型识别"软广/引流/情绪煽动"特征，降权 | 半自动 |
| **黑名单/白名单** | 人工维护的可信/不可信来源清单 | ✅ **老板的辅助正好用这里** |

## 2. 关键设计：定性标签 → 系统量化（解决老板"无法量化权重"的痛点）
老板说："我能打定性标签（律所实务>自媒体、某博主不可信），但无法量化权重。"
→ **解法**：老板只给**定性等级/黑名单**，由系统把定性映射为量化权重。例如：
```
来源等级 → 权重映射（系统内置，老板不用算）
  官方/法院/法规     → 1.0
  律所实务/专业机构   → 0.8
  主流媒体          → 0.5
  自媒体/论坛        → 0.3
  老板黑名单(某博主)  → 0.0（直接排除）
```
- 老板只需维护一个 `sources.yaml`：把来源归到某个等级，或拉黑某个具体博主/账号。
- 系统按映射自动加权 + 多源交叉验证 + Faithfulness 校验，**量化的活交给系统**。

## 3. 工厂落地（FB-5）
- 产出：`_factory/skills/data-quality.skill.md`（来源分级+交叉验证+去软文+忠实度校验流程）。
- 配套：每个项目一个 `sources.yaml`（老板可编辑的来源可信度/黑名单清单）。
- 复用：对接已配的 reranker/embedding 能力；忠实度校验用 GLM 做裁判。
