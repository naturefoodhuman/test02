# 技能：需求探索访谈（把模糊想法变成 DISCOVERY.md）

## 何时用
- DISCOVERY 阶段，老板给了一个想法但边界不清（FR-001-1/2）。

## 核心步骤
1. **澄清动机**：这东西解决谁的什么痛点？不做会怎样？
2. **挖核心场景**：让老板讲 ≥3 个具体使用场景（谁、在什么情况下、想得到什么）。
3. **主动补空白与矛盾**（FR-001-2）：列出他没说但必须定的点（数据从哪来？给谁用？要不要联网？）。
4. **调研同类**（FR-001-3）：搜同类开源项目、商业产品、**至少 1 个已知失败案例**。
5. **划分必须/以后**：MoSCoW（Must/Should/Could/Won't），帮他分清现在做什么。
6. **判断是否过大**：如果想法太大，建议一个更小的可行起点。
7. **草拟验收标准**（FR-001-4）：描述性的"什么算完成"，不写测试代码。

## 检查清单
- [ ] 核心用户场景 ≥ 3 个
- [ ] 同类产品/开源项目清单（≥3）
- [ ] 已知失败案例 ≥ 1 个
- [ ] 非功能要求（性能/安全/成本）
- [ ] 验收标准草案（描述性）
- [ ] 已主动指出至少 1 处空白或矛盾

## 反模式
- 直接开写需求文档而不先问。
- 把"想要的功能"当成"验收标准"。

## 输出物
- `docs/DISCOVERY.md`（Phase 1 Exit Artifact）
